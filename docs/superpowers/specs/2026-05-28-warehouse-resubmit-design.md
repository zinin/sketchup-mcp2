# Warehouse Resubmission — v0.2.0 Design

**Status:** spec — pending implementation plan
**Driver:** Extension Warehouse rejected v0.1.0 (`product_id=SU_MCP_SERVER`). Reviewer cited four blockers and two improvement notes. v0.2.0 addresses every cited issue.
**Out of scope:** Python PyPI package rename (`sketchup-mcp2` stays as-is — not part of the warehouse submission). Wire-protocol changes. Tool-API breaks visible to MCP clients.

## 1. Reviewer requirements → resolution map

| Reviewer line | Severity | Resolution |
|---|---|---|
| `eval` not allowed; if kept, disabled-by-default + warning + (preferably) per-call review | **Reject** | Disabled by default in the warehouse build; settings toggle with blocking `UI.messagebox` warning; Python side returns actionable text to the LLM when disabled. *Github-release build* ships with eval enabled (informed users, manual self-sign workflow). |
| Debug info in shared Ruby console clutters it | **Reject** | Default `log_level` lowered to `WARN`; every console line prefixed with `[MCPforSU]`; optional «Log to file» mode in Settings. |
| Name «SketchUp MCP Server» implies first-party | **Reject** | Rename to **«MCP Server for SketchUp»** (display) and **«MCPforSketchUp»** (Ruby module) — exactly the form reviewer suggested. |
| File prefix `su_` implies first-party | **Reject** | Directory tree and Ruby require-paths renamed to `mcp_for_sketchup/`. New Trimble `product_id` (`MCP_FOR_SKETCHUP`). |
| `model.start_operation` argument visible in Undo menu — should be human-readable | Note | All 10 mutating handlers use Title Case labels (e.g. `Create Component (Cube)`, `Boolean Union`, `Set Material (Red)`). |
| Silent rescues hide bugs | Note | All five identified silent rescues upgraded to explicit rescue + `Logger.log("DEBUG", ...)`. |

## 2. Versioning and breaking-change policy

This is treated as a clean break: we are the only users of v0.1.0, so no migration code is written.

- Bump both halves to **0.2.0** in seven canonical locations (per `docs/release.md` §1).
- `compat.py::MIN_RUBY = "0.2.0"`, `compat.rb::MIN_PYTHON = "0.2.0"` — old clients/extensions intentionally reject each other at handshake. Old `0.1.0` `.rbz` does not coexist; users must re-install.
- Trimble `product_id`: `SU_MCP_SERVER` → `MCP_FOR_SKETCHUP`. Submit as a *new product* in the warehouse intake form; the old `SU_MCP_SERVER` listing stays as a dead record at Trimble. If the intake form rejects the new ID (e.g. requires linking to a prior submission), surface to user before pushing — do not silently re-use the old ID.
- SketchUp preferences `SECTION` is renamed without migration. First launch after upgrade reads defaults. (For us, the dev machine sees a one-time `host/port/log_level` reset — acceptable.)

Module choice rationale: `MCPforSketchUp` is a valid Ruby constant (starts with capital), short enough that fully-qualified references like `MCPforSketchUp::Handlers::Geometry` stay readable, and mirrors the display name's word boundaries. We deliberately do not use a long SHOUT_SNAKE form (`MCP_FOR_SKETCHUP`) — it reads worse in handler files where the module appears repeatedly.

Console prefix `[MCPforSU]` is the camel-case short form; the alternative `[mcp_for_sketchup]` is verbose for every log line. Choice is purely a readability call; if reviewers object during re-review, we can swap in a one-character edit to `core/logger.rb`.

## 4. eval_ruby gate

### 4.1 Build-time variant

`package.rb` accepts `--variant=warehouse|github` (default: warehouse). At pack time it generates a single file `mcp_for_sketchup/mcp_for_sketchup/core/build_profile.rb`:

```ruby
# AUTOGENERATED by package.rb — do not edit. Regenerated per build.
module MCPforSketchUp
  module Core
    module BuildProfile
      VARIANT                 = "warehouse"  # or "github"
      EVAL_ENABLED_BY_DEFAULT = false        # or true
    end
  end
end
```

`build_profile.rb` is in `.gitignore`. Tests run without it; in its absence `Config.eval_enabled_default` returns `false` (matches warehouse — the safer default for tests).

Two artifacts are produced:
- `mcp_for_sketchup_v0.2.0-warehouse.rbz` — for Trimble Extension Warehouse submission.
- `mcp_for_sketchup_v0.2.0-github.rbz` — for GitHub Releases (self-signed via Trimble's signing service; eval enabled).

### 4.2 Runtime gate

- **Sentinel-`nil` mechanism for the unset-pref state.** `Config::DEFAULTS[:eval_enabled]` is `nil` (NOT `false`). `load_from_defaults!` calls `read_default(SECTION, "eval_enabled", nil)` and assigns `self.eval_enabled = raw_eval.nil? ? nil : !!raw_eval` so that an explicit `false` is preserved as `false` but an absent pref leaves the instance variable `nil`. (This relies on SketchUp's `read_default` returning the default arg when the key is absent and the persisted value when present, including boolean `false`. The plan's CRITICAL-1 test exercises both branches; see also QUESTION-7 in iter-1 review.)
- `Config.eval_enabled?` returns `@eval_enabled` when non-nil; when `@eval_enabled` is `nil`, falls through to `Core::BuildProfile::EVAL_ENABLED_BY_DEFAULT` when that constant is defined, otherwise returns `false` (safe warehouse default for tests / dev runs without build_profile).
- Settings UI and all other callers MUST consult `Config.eval_enabled?`, never the raw `Config.eval_enabled` accessor — see settings_dialog `on_load_state` which uses `eval_enabled?` to populate the checkbox.
- `handlers/eval.rb::eval_ruby` checks the gate first:

```ruby
def self.eval_ruby(params)
  unless Core::Config.eval_enabled?
    raise Core::StructuredError.new(-32010,
      "eval_ruby is disabled. Open Plugins → MCP Server → Settings... and " \
      "check 'Enable Ruby evaluation'. WARNING: this grants the MCP server " \
      "arbitrary code execution including filesystem and shell access.")
  end
  code = V.require_string(params, "code")
  # ... existing eval body ...
end
```

The `-32010` code is JSON-RPC application-error range (`-32000..-32099`).

### 4.3 Settings UI for the gate

Settings dialog adds one checkbox `Enable Ruby evaluation (DANGEROUS)`. When the user *transitions from off to on* and clicks Save, a blocking `UI.messagebox` appears:

> Enabling «Ruby evaluation» lets connected MCP clients execute arbitrary Ruby code inside SketchUp with FULL access to your filesystem, network, and shell. Only enable this with MCP clients you fully trust. Continue?
>
> [No] [Yes]

**Two-phase confirmation flow (required to avoid the Windows `UI.messagebox`-inside-callback hang already documented at `settings_dialog.rb:100`):**

1. `on_save` action_callback validates the payload. If the eval transition is detected (off→on), it does NOT show the messagebox synchronously. Instead it defers via `::UI.start_timer(0, false)` so the callback frame is left first.
2. The deferred block shows the messagebox.
3. On **Yes**: persist via `Config.update!`, then `applyState(saved + eval_enabled: true)` to confirm the new state in the dialog.
4. On **No**: do NOT persist. Call `applyState(saved)` (which sets the checkbox back to its previously-saved value) and write a one-line error into the new `<div id="eval_enabled-error">` element ("Cancelled — Ruby evaluation remains disabled"). The dialog's `clearErrors` helper MUST include `eval_enabled` in its array so the message is cleared on the next interaction.
5. **Timer-internal rescue (iter-2 CRITICAL-2).** The deferred block runs in
   its own `begin/rescue StandardError` — exceptions raised inside the
   timer callback do NOT propagate to `on_save`'s outer rescue (the timer
   fires after the original action_callback frame has returned). On
   exception inside the timer body, log via `Logger.log_error("settings_dialog.eval_confirm", e)`,
   then `applyState(saved)` + `onSaveResult({ok:false, errors:{_general:…}})`
   so the dialog reverts and the user sees an actionable message instead
   of a silently-broken state.
6. **Shared persist/restart helper (iter-2 CRITICAL-2).** Both the normal
   path and the confirm-Yes branch run identical persist + host/port
   restart-prompt logic. Extract a `persist_and_finalize` helper called
   from both sites so the Yes-branch does not miss the
   restart-needed check or `dialog.close` deferral.

This pattern mirrors the existing host/port restart prompt (which also uses `::UI.start_timer(0, false)`).

### 4.4 Python side surface

`src/sketchup_mcp/tools.py::eval_ruby` catches `SketchUpError` with `code == -32010` and returns the error's message as the tool result string (not a raise). The LLM receives an actionable text and surfaces it to the user verbatim. Centralising this in `_call` is undesirable — only `eval_ruby` should special-case the disabled-state.

Example flow:
1. User: «build me a cabinet via eval_ruby»
2. LLM calls `eval_ruby(code=...)`.
3. Ruby returns `error: {code: -32010, message: "eval_ruby is disabled..."}`.
4. Python `eval_ruby` returns the message as a normal string result.
5. LLM speaks the message to the user.
6. User opens Settings, enables, confirms the messagebox warning.
7. Subsequent `eval_ruby` calls succeed.

## 5. Logging strategy

### 5.1 Defaults and prefix

- `Config::DEFAULTS[:log_level]` flips from `"INFO"` to `"WARN"`.
- The `[MCPforSU]` prefix is applied in a new shared low-level private method `Logger._emit` — NOT only inside `Logger.log`. This is required so that backtrace lines emitted by `Logger.log_error` (which calls `_emit("    #{bt}")` directly per line) also carry the prefix. Acceptance §12.7 («every console line includes `[MCPforSU]`») would otherwise fail on error paths.
- `Logger._emit` builds `"[#{timestamp}] [MCPforSU] [#{level}] #{msg}"` for level-tagged lines, and `"[MCPforSU]     #{bt}"` for backtrace continuation lines (timestamp omitted because callers may emit them inside larger frames).
- **All existing public Logger methods are preserved verbatim (iter-2 CRITICAL-1).** `log`, `log_tool` (used by `application.rb` + `server.rb`, 9 call sites), and `log_error` keep their signatures and gating. The only change is that they route through `_emit` instead of writing directly.
- **Backtrace gating preserved (iter-2 CRITICAL-1).** `log_error` continues to emit backtrace lines only when `Config.log_level == "DEBUG"`, capped at `first(3)` — the current behaviour. Rewriting it to emit full backtrace at every level would violate reviewer rejection 2 («Debug info clutters shared Ruby console»).

### 5.2 Optional log-to-file

Two new prefs:

| Pref key | Type | Default | Constraint |
|---|---|---|---|
| `eval_enabled` | boolean / nil | `nil` (unset → falls back to `BuildProfile::EVAL_ENABLED_BY_DEFAULT`) | — |
| `log_to_file` | boolean | `false` | — |
| `log_file_path` | string | `File.join(Dir.tmpdir, "mcp_for_sketchup.log")` | When `log_to_file=true`: non-empty AND `File.directory?(File.dirname(File.expand_path(path)))` (parent dir must already exist — we do not auto-create user-facing log directories). Validator enforces both checks. |

`Logger.write`:
- Always emits to `SKETCHUP_CONSOLE` (or `$stdout` in tests) — unchanged behaviour.
- *Additionally* appends to `log_file_path` when `Config.log_to_file?` is true.
- File-IO errors swallowed silently with a one-time `DEBUG` console line ("log file write failed: …; reverting to console only"). We do not raise — logging must never break the data path.

`Dir.tmpdir` rather than `~/Documents/...` because: cross-platform (works identically on macOS/Windows/Linux), no mkdir-in-Documents permission edge cases, and a logfile in tempdir matches user expectation for a developer-mode toggle. If the user wants a stable path, they edit `log_file_path` in Settings.

### 5.3 Show Log menu

Currently `Plugins → MCP Server → Show Log` calls `SKETCHUP_CONSOLE.show`. New behavior:

- If `log_to_file=true` and the file exists → open in the OS default viewer via a small `file_uri_for(path)` helper (iter-2 CRITICAL-3):
  ```ruby
  def self.file_uri_for(path)
    encoded = File.expand_path(path).gsub('\\', '/')
    encoded = "/#{encoded}" if encoded =~ /\A[A-Za-z]:/   # Windows drive letter
    encoded = URI::DEFAULT_PARSER.escape(encoded)
    "file://#{encoded}"
  end
  # …
  UI.openURL(file_uri_for(log_file_path))
  ```
  `URI::File.build(path: …)` is NOT used — it raises `URI::InvalidComponentError` on paths containing spaces or non-ASCII characters under MRI 3.2.x (verified by iter-2 codex review). The escape helper above is portable: it normalises Windows backslashes to forward slashes, prepends the leading slash for `C:` drive-letter paths required by RFC 8089, then URI-encodes the result before string-interpolating into the `file://` scheme.
- Otherwise → existing `SKETCHUP_CONSOLE.show`.

## 10. Out-of-scope decisions (deliberate)

- **PyPI name `sketchup-mcp2`.** Stays. Not part of warehouse submission; renaming would break PyPI continuity.
- **Python module `sketchup_mcp`.** Stays. Not reviewer-flagged.
- **Menu label «MCP Server».** Stays. Reviewer's complaint was the extension *name*, not the menu label. A submenu literally named «MCP Server» is unambiguous once the extension display name properly attributes the author.
- **Migration code for old `SU_MCP` prefs.** Skipped; we are the only users of 0.1.0.
- **Per-call eval_ruby confirmation dialog.** Rejected as a default — would break LLM-driven multi-step workflows that are the primary use-case. Could be added in a future version as an opt-in «paranoid mode» pref.

## 11. Risks and verification

- **Trimble re-rejection on a missed item.** Mitigation: cross-check every rejection line in §1 against the implementation before re-submission. Maintain a release checklist appended to `docs/release.md`.
- **product_id change confusing Trimble's intake.** Mitigation: treat as a new submission, not an update. Surface to the user before clicking the final submit button if the intake form behavior is ambiguous — do not auto-link to the dead `SU_MCP_SERVER` record.
- **CLAUDE.md / README references missed in renaming.** Mitigation: a `git grep -in 'su_mcp\|SU_MCP\|Sketchup MCP Server'` should return zero matches after the rename pass.
- **Tests passing locally but broken in `.rbz`.** `package.rb` generates `build_profile.rb` at pack time; verify by unzipping the produced `.rbz` and confirming the file's contents.
- **eval_ruby pref persisting across the two variants.** A user who installs github-variant first (eval pref → true) and then upgrades to warehouse-variant of same product_id keeps `eval_enabled=true` in their prefs. This is acceptable: warehouse-only users won't have the github build; cross-installers are technical and understand the toggle.

## 12. Acceptance criteria

1. `ruby package.rb --variant=warehouse` produces `mcp_for_sketchup_v0.2.0-warehouse.rbz` with `EVAL_ENABLED_BY_DEFAULT = false` baked in.
2. `ruby package.rb --variant=github` produces `mcp_for_sketchup_v0.2.0-github.rbz` with `EVAL_ENABLED_BY_DEFAULT = true`.
3. `ruby test/run_all.rb` passes (existing 230 tests + new operation/eval/logger/config tests).
4. `uv run pytest tests/ -q` passes (existing 120 + new eval-error-routing test).
5. `git grep -in 'su_mcp\|SU_MCP\|Sketchup MCP Server' -- ':!docs/superpowers/specs/*' ':!docs/session-transfer-*'` returns no matches.
6. Fresh install of warehouse-variant, server started, Claude calls `eval_ruby` → user sees actionable text. Enable in Settings (confirms blocking messagebox) → next `eval_ruby` call succeeds.
7. With `log_to_file=false`, every console line includes `[MCPforSU]`. With `log_to_file=true`, the same lines also appear in `Dir.tmpdir/mcp_for_sketchup.log`.
8. Performing `Create Component (Cube)` then `Edit → Undo` in SketchUp shows the title-cased label in the Undo menu before undo, and removes the cube on undo.
